# ARIS Product Vision

## 1. 제품 정의

ARIS는 `happy`의 세션 동기화/암호화/에이전트 브리지 역량을 기반으로, **웹을 중심으로 모바일/태블릿/데스크톱을 아우르는** Agentic Runtime Interface입니다.

핵심 목표는 어떤 디바이스에서도 실제 운영 오케스트레이션(실행/중단/승인/상태 분석)을 수행하면서, UI/UX는 모던 생산성 앱 수준의 심미성과 친숙성을 제공하는 것입니다.

## 2. 문제 정의

현재 `happy` 기반 흐름은 강력하지만 다음 한계가 있습니다.

- 다중 에이전트 세션을 비교/우선순위 관리하기 어려움
- 권한 요청/도구 실행 이벤트를 "로그"로는 보지만 "행동 가능한 UI"로 관리하기 어려움
- 모바일/태블릿에서 실질적인 세션 제어 UX가 부족함
- AI 응답 타입(텍스트/명령 실행/코드 읽기/코드 작성)에 따라 정보 표현이 충분히 구분되지 않음

## 3. ARIS 목표

- 모던 앱 감각의 UI/UX: 화이트 베이스 + 고품질 컴포넌트 시스템
- 멀티 에이전트 운영: claude/codex/gemini를 동일 모델로 제어
- 크로스디바이스 UX: 모바일/태블릿/데스크톱에서 동일한 핵심 액션 제공
- 응답 타입 기반 렌더링: AI 응답 종류별 맞춤 UI 컴포넌트 제공
- 의도 전달 UX 강화: 사용자 입력/액션 컴포넌트를 시각적으로 명확하게 제공
- 보안 강화: 로그인 기반 접근통제 + 종단간 암호화(E2E) 모델 확립
- 유연성 보장: UI로 처리되지 않는 경우를 위한 SSH 직접 접근 옵션 제공

## 4. Non-Goals (초기 범위 제외)

- `happy-server` 프로토콜 전면 변경
- 새로운 에이전트 프로세스 엔진 구현
- 완전한 IDE 기능(코드 에디터 대체)

## 5. 타겟 사용자

- 1차: 서버에서 에이전트를 상시 실행하는 개인/소규모 팀 개발자
- 2차: 여러 세션을 병렬 운영하는 운영형 개발자(에이전트 오퍼레이터)
- 3차: 이동 중(모바일/태블릿) 승인/중단/재시작을 처리해야 하는 사용자
- 4차: 노션 등 모던 앱 인터랙션에 익숙한 일반 프로덕트 사용자

## 6. 성공 지표 (MVP)

- TTR(Time To Respond): permission 요청 확인까지 평균 10초 이내
- 모바일 운영성: 모바일에서 핵심 액션(approve/deny/abort/retry) 3탭 이내
- 응답 가독성: 응답 타입 구분 인지율(사용성 테스트) 90%+
- 사용자 만족도: UI 친숙성/심미성 내부 평가 4.5/5 이상
- 보안 기준: 인증 없는 접근 0건, 민감 이벤트 감사로그 100% 기록

## 7. 제품 원칙

- Familiar Modern UX: 노션류 모던 앱에 익숙한 사용자에게 학습 비용이 낮아야 함
- White-first Visual System: 기본 배경은 화이트로 유지하고 컴포넌트 컬러로 의미를 전달
- Semantic Color Language: 색상은 장식이 아니라 상태/행동/응답 타입을 설명해야 함
- Cross-Device Consistency: 디바이스가 달라도 핵심 정보구조와 액션은 동일해야 함
- Compatibility by Default: `happy`와 최대 호환, 최소 침습
- Zero-Trust Access: 로그인 없는 접근을 허용하지 않으며, 최소권한 원칙 적용
- Secure by Inheritance+: `happy`의 E2E 암호화 모델을 계승하고 ARIS 인증/권한 체계를 추가

## 8. 중장기 확장 방향

- 시각화 기반 Agent Skills 관리
- 시각화 기반 지침 문서(`AGENTS.md`) 관리
- 프로젝트별 에이전트 실행 이력/결과 시각화
- 프로젝트별 문서화(Documentation) 관리
