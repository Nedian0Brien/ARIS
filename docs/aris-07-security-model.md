# ARIS Security Model

## 1. 목표

- 인증 없는 접근 차단
- 최소 권한 원칙 기반 런타임 제어
- 데이터 기밀성/무결성 보장
- 운영 액션의 감사 가능성 확보
- SSH fallback을 안전하게 통제

## 2. 위협 모델 (요약)

- 익명 사용자의 무단 접근
- 탈취된 토큰을 통한 세션 조작
- 네트워크 구간 데이터 노출
- 서버 저장 데이터 유출 시 민감 정보 노출
- 운영자 오조작 또는 악의적 액션
- SSH fallback 채널을 통한 무단 원격 제어

## 3. 인증/세션 정책

### 로그인 필수

- `/login` 인증 성공 전에는 런타임 라우트 접근 불가
- API/WS 모두 토큰 검증 실패 시 즉시 차단

### 토큰 정책

- Access Token: 짧은 수명(예: 15분)
- Refresh Token: 회전(rotation) 적용, 탈취 대응
- 토큰 저장: 브라우저 저장소 직접 보관 최소화(가능하면 secure cookie)

### 장치/세션 통제

- 장치별 세션 식별자 관리
- 의심 로그인(새 장치/이상 위치) 시 재인증
- 로그아웃 시 refresh token 폐기

## 4. 권한 모델

- `Operator`: 승인/중단/재시작/SSH fallback 제어 가능
- `Viewer`: 조회 전용
- 민감 액션(예: Kill, 정책 변경, SSH 접속)은 재확인 단계 적용

## 5. 암호화 정책

### 전송 구간

- HTTPS/TLS 강제
- WS는 WSS로만 허용
- SSH fallback 웹 채널도 TLS 하에서만 허용

### 저장/메시지 구간

- `happy`의 E2E 암호화 모델 계승
- 세션 메시지/메타데이터는 클라이언트 측 암호화 우선
- 서버는 가능하면 평문 대신 암호문 저장

### 키 관리

- 키는 코드/저장소에 하드코딩 금지
- 운영 환경에서는 비밀 저장소(Secret Manager/Vault) 사용
- 키 교체 주기 및 비상 폐기 절차 정의

## 6. SSH Fallback 보안 정책

- 기본값: 비활성
- 활성 조건: `Operator` 권한 + 명시적 정책 허용
- 접속 방식: 사용자별 단기 세션 토큰 + 세션 TTL
- 제한 정책: 허용 호스트/포트/명령 제한(가능한 범위)
- 감사 항목: 접속 시작/종료, 실행 명령, 세션 길이, 사용자 식별자
- 비상 차단: 정책 토글로 전체 SSH fallback 즉시 중단 가능

## 7. 감사 및 탐지

- 감사 이벤트 필수 수집:
  - 로그인/로그아웃/재인증
  - permission 승인/거절
  - 세션 제어(Abort/Retry/Kill/Resume)
  - SSH fallback 접속/종료/명령 실행
  - 정책 변경
- 감사로그는 변조 방지 가능한 저장소에 보관
- 이상 징후(실패 로그인 급증, 대량 중단, 비정상 SSH 접속) 알림 연계

## 8. 보안 체크리스트 (MVP)

- 로그인 없는 접근이 차단되는가
- API/WS 모두 인증 강제되는가
- 민감 액션이 감사로그로 남는가
- TLS가 강제되는가
- 토큰 만료/갱신/폐기 플로우가 동작하는가
- E2E 암호화 경계가 문서와 구현에서 일치하는가
- SSH fallback 권한/세션 만료/감사 정책이 동작하는가

## 9. 운영 가이드 (초안)

- 정기 점검: 토큰 수명, 키 회전, 감사로그 적재 상태
- 배포 점검: HTTPS 인증서, 보안 헤더, 비밀키 주입 방식
- SSH 점검: 허용 대상, 정책 상태, 비상 차단 스위치
- 사고 대응: 토큰 폐기, 키 롤오버, 영향 범위 추적 순서 문서화
