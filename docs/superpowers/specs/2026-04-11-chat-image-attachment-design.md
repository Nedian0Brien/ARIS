# 채팅 이미지 첨부 전달 기능 설계

## 목표

채팅 화면의 `+` 버튼에서 이미지를 업로드하고, 업로드된 이미지를 서버 자산으로 저장한 뒤 에이전트에게 해당 이미지의 서버 경로와 함께 참고 대상으로 전달한다. 사용자 화면에서는 전송 전 composer와 전송 후 사용자 채팅 버블 모두에서 첨부 이미지를 자연스럽게 확인할 수 있어야 한다.

## 범위

- 채팅 composer의 `+` 메뉴에 `사진 업로드` 액션 추가
- 로컬 이미지 파일 업로드 API 및 서버 자산 저장 경로 추가
- composer 상태에 `image` 첨부 항목 추가
- 사용자 메시지 전송 시 이미지 첨부 메타(`attachments`) 전달
- 사용자 채팅 버블 상단에 첨부 이미지 프리뷰 렌더링
- 업로드 실패, 프리뷰 로드 실패, 첨부 제거 흐름 처리

## 비범위

- 이미지 OCR/자동 캡션 생성
- 이미지 확대 모달, 이미지 편집, 드래그 앤 드롭 업로드
- PDF, 동영상, 일반 바이너리 파일까지 포함하는 범용 자산 시스템 확장
- 에이전트별 별도 멀티모달 API 연동

## 현재 구조

- composer의 `+` 메뉴는 `파일 첨부`와 `텍스트 추가`만 지원하며, 상태는 `services/aris-web/app/sessions/[sessionId]/ChatInterface.tsx`에서 관리한다.
- 현재 컨텍스트 항목은 사실상 텍스트 기반이며, 전송 시 최종적으로 문자열 프롬프트로 합쳐진다.
- 메시지 전송은 `services/aris-web/app/api/runtime/sessions/[sessionId]/events/route.ts`를 통해 이뤄지며, `text`와 `meta` 중심 구조다.
- 채팅 버블 렌더링도 같은 `ChatInterface.tsx`에서 처리하고 있다.

## 설계

### 1. 업로드 자산 모델

- 이미지 업로드는 메시지 전송과 분리한다.
- 사용자가 이미지를 선택하면 즉시 업로드 API를 호출해 서버에 저장하고, 성공 시 아래 메타데이터를 반환한다.
  - `assetId`
  - `name`
  - `mimeType`
  - `size`
  - `width`
  - `height`
  - `serverPath`
  - `previewUrl`
- `serverPath`는 에이전트에게 전달되는 기준 경로이며, `previewUrl`은 사용자 UI 렌더링용 경로다.
- 저장 위치는 서버가 읽고 관리할 수 있는 전용 디렉터리로 고정한다. Git 추적 대상이나 workspace 일반 파일 목록과 섞이지 않게 분리한다.

### 2. Composer UI

- `+` 메뉴에 `사진 업로드` 항목을 추가한다.
- 업로드 성공 후 composer에는 텍스트 입력 영역 근처에 첨부 이미지 카드/칩을 표시한다.
- 카드/칩은 아래 정보만 보여준다.
  - 썸네일
  - 파일명
  - 삭제 버튼
  - 업로드 중 또는 실패 상태
- 시각 스타일은 절제된 보더와 작은 그림자 수준으로 제한한다.
- 강한 그러데이션, 과도한 배경 강조, 장황한 보조 설명은 넣지 않는다.

### 3. 메시지 전송 방식

- 실제 메시지 전송 시 현재의 `text` 기반 API는 유지한다.
- 다만 전송 직전 최종 프롬프트 앞에 이미지 참조 블록을 붙인다.
- 예시 형식:

```text
<image_attachment assetId="..." serverPath="..." mimeType="image/png">
첨부 이미지를 참고해서 답변하라.
</image_attachment>
```

- 첨부 이미지가 여러 장이면 블록을 순서대로 누적한다.
- 사용자 본문 텍스트와 첨부 메타는 분리해 관리한다.

### 4. 메시지 메타 구조

- UI 렌더링은 프롬프트 문자열을 다시 파싱하지 않고 `meta.attachments`만 사용한다.
- `attachments`는 아래 구조를 기준으로 둔다.

```ts
type ChatImageAttachment = {
  assetId: string;
  kind: 'image';
  name: string;
  mimeType: string;
  size: number;
  width?: number;
  height?: number;
  serverPath: string;
  previewUrl: string;
};
```

- 사용자 메시지 전송 시 `meta.attachments`에 첨부 배열을 포함한다.
- 버블 렌더링 시에는 이 메타를 사용해 썸네일, 파일명, fallback 상태를 표시한다.

### 5. 채팅 버블 렌더링

- 사용자 메시지에 `attachments`가 존재하면 채팅 버블 상단에 이미지 프리뷰 strip을 먼저 렌더링한다.
- 그 아래에 기존 텍스트 본문을 표시한다.
- 단일 이미지는 큰 카드 1개, 다중 이미지는 일정 크기의 썸네일 목록으로 처리한다.
- 프리뷰 이미지 로드 실패 시에는 빈 박스 대신 파일명과 첨부 배지를 표시한다.
- 향후 확대 보기 기능은 추가할 수 있지만 이번 범위에서는 제외한다.

### 6. 실패 처리

- 업로드 실패:
  - composer 안에서 즉시 실패 상태를 표시한다.
  - 실패한 항목은 전송 대상에서 제외하거나 재시도/삭제할 수 있어야 한다.
- 전송 전 첨부 제거:
  - composer 카드/칩의 삭제 버튼으로 제거한다.
- 프리뷰 로드 실패:
  - 버블에서는 첨부가 존재함을 유지하되 이미지 대신 fallback UI를 노출한다.
- 서버 저장은 성공했지만 프리뷰만 실패한 경우:
  - 에이전트 전달은 계속 진행한다.

## API 방향

- 신규 업로드 API
  - 예: `POST /api/runtime/sessions/:sessionId/assets/images`
  - multipart 업로드를 받아 서버 자산으로 저장하고 메타데이터 반환
- 기존 메시지 전송 API
  - `POST /api/runtime/sessions/:sessionId/events`
  - `meta.attachments` 허용

## 시각 가이드

- 첨부 UI는 눈에 띄어야 하지만 본문보다 튀면 안 된다.
- 기본 원칙:
  - 작은 radius
  - 약한 보더
  - 절제된 그림자
  - 명확한 썸네일
  - 짧은 상태 문구
- 채팅 전체 톤과 충돌하는 장식 요소는 피한다.

## 테스트 계획

- 최소 테스트 범위:
  - 이미지 업로드 성공 시 composer에 첨부 카드가 나타남
  - 첨부 제거 시 composer 상태에서 사라짐
  - 첨부 포함 전송 시 `meta.attachments`가 함께 전송됨
  - 사용자 채팅 버블에 첨부 이미지 프리뷰가 렌더링됨
  - 업로드 실패 시 오류 상태가 노출됨
  - 프리뷰 로드 실패 시 fallback UI가 렌더링됨
  - 다중 이미지 첨부 시 순서가 유지됨

## 구현 순서

1. 이미지 자산 업로드 API와 저장 경로 설계
2. composer의 `image` 첨부 상태 및 업로드 UI 추가
3. 메시지 전송 payload에 `attachments` 메타 연결
4. 사용자 버블 첨부 프리뷰 렌더링 추가
5. 실패/fallback 처리
6. 관련 테스트 및 타입 점검

## 리스크

- 서버 자산 저장 위치를 잘못 잡으면 배포 환경에서 권한 또는 정리 정책 이슈가 생길 수 있다.
- 버블 프리뷰를 문자열 파싱 기반으로 구현하면 이후 유지보수가 빠르게 나빠진다. 반드시 `meta.attachments` 기반으로 고정해야 한다.
- `ChatInterface.tsx`가 이미 큰 파일이므로, 구현 과정에서 첨부 관련 렌더링/헬퍼를 적절히 분리할지 검토가 필요하다.
