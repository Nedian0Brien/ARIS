# ARIS Design Thinking

## 1. Purpose

### 타겟 사용자

- 서버에서 `happy` daemon과 CLI 세션을 상시 운용하는 개발자
- 이동 중 모바일/태블릿에서 빠르게 운영 액션을 수행해야 하는 사용자
- 노션 등 모던 앱의 정보구조와 인터랙션에 익숙한 사용자

### 핵심 문제

- 런타임 이벤트가 산재되어 있어 운영 결정을 빠르게 내리기 어렵다.
- AI 응답이 타입별로 명확히 구분되지 않아 가독성과 행동 결정이 느리다.
- 소형 화면에서 로그 탐색과 승인/중단 액션이 번거롭다.
- UI로 처리되지 않는 예외 상황에서 즉시 대처 수단이 부족하다.

### 성공 시나리오

1. 대시보드 진입 후 5초 내 상태 파악
2. 응답 타입별 UI로 현재 상황을 즉시 이해
3. 의도 전달 컴포넌트로 빠르게 지시/승인/취소 수행
4. 예외 상황에서 SSH 직접 접근으로 즉시 복구

## 2. Tone

### 선택 톤

**Modern Productive, Friendly Professional**

### 톤 가이드

- "관제 콘솔"보다 "모던 프로덕션 앱"에 가까운 친숙한 밀도
- 기본 배경은 화이트/라이트 뉴트럴
- 핵심 정보/행동은 다채로운 의미 기반 컬러로 강조
- 텍스트 가독성과 공간 질서를 유지하면서도 시각적 즐거움을 제공

## 3. Differentiator

### 기억에 남을 요소

**Color-Coded Agentic Blocks**: AI 응답과 사용자 인터랙션을 타입별 컬러 블록/아이콘 컴포넌트로 표현한다.

### 차별 기능

- Response Blocks: 텍스트/명령 실행/코드 읽기/코드 작성 응답을 타입별 UI로 렌더링
- Intent Composer: 사용자가 의도를 전달하는 액션 칩, 모드 버튼, 프리셋 프롬프트
- Contextual Action Bar: 승인/재시도/중단/복구 액션을 문맥별로 강조
- SSH Fallback Entry: UI 범위를 벗어나는 경우 즉시 SSH 콘솔로 전환

## 4. Constraints

### 기술 제약

- 백엔드: `happy-server` API/WebSocket 경계 준수
- 런타임: `happy-cli` + daemon 기반 세션 제어 모델 유지
- 배포: 기존 도메인 + nginx reverse proxy 구조에 탑재 가능해야 함

### 품질 제약

- 초기 로드 3초 이내(일반 서버 환경 기준)
- 동시 세션 5~10개에서 UI 프레임 저하 없이 동작
- 응답 타입별 시각 구분이 일관되고 명확해야 함
- 모바일 환경에서도 핵심 액션 3탭 이내 수행 가능

### 접근성/반응형 제약

- 키보드 중심 운영 지원 (상태 이동/승인/취소)
- 색상 외 텍스트/아이콘으로 상태 중복 표현
- 최소 WCAG AA 대비 기준 유지
- 터치 타겟 최소 44x44px
- Breakpoint별 레이아웃 전환 기준 명시 (mobile/tablet/desktop)

### 보안 제약

- 로그인 전에는 어떠한 런타임 데이터도 노출하지 않음
- 민감 데이터는 전송/저장 시 암호화 적용
- SSH fallback 접근은 권한/감사 정책 하에서만 허용

---

Design Thinking 결론: ARIS는 "웹 기반 Cross-Device Agentic Workbench"로 설계하며, 화이트 베이스 위에 의미 기반 컬러 컴포넌트로 정보와 인터랙션을 전달한다.
