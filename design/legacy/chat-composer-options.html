현재 composer는 "괜찮지만 평범"합니다. 이 파일은 세 가지 서로 다른 철학에서 출발한 개선안을 제시합니다. A는 현재 구조를 정제하고, B는 밀도를 극대화하며, C는 모드에 반응하는 적응형입니다. 이 문서에서 하나를 골라 chat-screen-v1.html에 반영합니다.

00 · Reference

현재 composer (chat-screen-v1)

개선 전 기준점입니다. 첨부 chip · textarea · 하단 툴바(+/📎/@ 아이콘 · 모델 selector · Send). 문제는 없지만 Linear 수준 폴리시로 보면 단순한 박스에 그칩니다.

!시각적 위계가 평평함

"Send"가 가장 중요한 액션인데 다른 툴들과 동급으로 보입니다. 액션 중심성이 약합니다.

!모델 selector가 산만함

버튼처럼 강조되어 있어 매 메시지 전에 눈이 거기로 갑니다. 대부분의 경우 바꾸지 않는 설정임에도.

!모드 구분이 없음

질문 · 계획 · 실행 · 터미널이 모두 같은 인풋을 쓰지만 의도는 다릅니다. 인풋이 의도에 반응하지 않습니다.

A
Option A · Focused MinimalLow risk

현재 구조를 정제한다.

철학: "Send가 주인공이다." 레이아웃은 그대로. 대신 Send 버튼을 pill 형태로 키우고, 모델 selector는 subtle chip으로 강등해 매 순간 눈이 가지 않게 합니다. 경계선을 줄여 전체가 한 장의 카드처럼 보이게 만듭니다.

05 · Recommendation

추천: A를 베이스로 C의 모드를 점진적으로.

My pick

Option A + Option C 모드 하이브리드

먼저 A(Focused Minimal)로 기본 틀을 확정합니다. Send pill, subtle model chip, 내부 경계선 제거 — 이 세 가지만으로도 톤이 눈에 띄게 Linear 급이 됩니다. 그 위에 C의 모드 탭을 optional로 얹습니다. 기본은 Ask mode(지금과 동일한 경험), 사용자가 ⌘⇧M 또는 상단 작은 버튼으로 Plan/Agent/Terminal로 명시적 전환. Send 버튼 컬러만 모드에 반응하는 수준에서 시작해, Plan/Terminal 기능이 준비되는 시점에 placeholder·tools까지 확장.

B의 rail은 포기합니다. 매력적이지만 ARIS의 모바일 비중을 고려할 때 수직 공간 효율이 떨어지고, 모바일 대응 비용이 큽니다. rail에 넣고 싶었던 도구들은 A의 + 플러스 메뉴에 수용 가능합니다.

결론 — Option A를 즉시 적용, C 모드는 느슨한 phase 2.

1Phase 1 — chat-screen-v1에 즉시 반영

Option A의 3가지 상태(empty / focus / running)를 chat-screen-v1.html의 composer 섹션에 덮어씁니다. 토큰 기반이므로 이후 tokens.css 마이그레이션과 자연스럽게 연결됩니다.

2Phase 2 — 모드 탭 도입 (optional)

Plan/Terminal이 실제 기능으로 준비되는 시점에 Option C의 모드 탭만 얹습니다. Send 버튼 컬러는 모드에 반응, placeholder/tool은 모드별로 다르게.
