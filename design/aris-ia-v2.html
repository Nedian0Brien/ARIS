IA Proposal · v2 · Draft

ARIS를 네 개의 표면으로
재구성합니다.

현재의 "Workspace → Chat" 일관 경로를 해체하고, 사용자의 의도에 따라 Home · Ask ARIS · Project · Files 네 개의 진입점을 병렬로 제공합니다. 용어도 손봐, 디렉토리에 1:1로 묶이는 단위는 Project로 통일합니다.

01 · Why this change

지금의 IA가 작업을 강제한다.

모든 경로가 "일단 Workspace를 하나 골라야" 시작됩니다. 장기 프로젝트엔 맞지만, 일회성 질문 · 교차 비교 · 서버 점검처럼 Workspace 프레임 바깥의 작업엔 매번 불편한 전환을 강요합니다.

현재 구조

모든 채팅은 필수로 어떤 Workspace에 속하고, Workspace는 로컬 디렉토리에 1:1로 대응합니다. 좌측 사이드바는 Workspace 내부 채팅을 시간 그룹(Pinned / Today / Yesterday / Earlier)으로 나눠 보여줍니다.

문제점 4가지

1. 질문 하나 던지려면 먼저 Workspace부터

LLM 사용의 절반 이상은 프로젝트 독립적인 가벼운 질문. 그런데 ARIS에선 임시 Workspace를 만들거나 관계없는 곳에 얹어야 해서 맥락이 섞입니다.

2. 프로젝트 전체 뷰가 없다

한 프로젝트의 누적 채팅 · 파일 변경 · 컨텍스트 자산을 한눈에 보는 진입점이 없음. 결정과 자산이 채팅 속에만 녹아 있습니다.

3. 서버 · 파일 작업이 Workspace 하위에 갇혀 있다

로그 조회 · 디스크 확인 · 프로젝트 밖 경로 탐색도 결국 특정 Workspace 안에서 터미널을 열어야. 크로스-프로젝트 작업이 어색해집니다.

4. "Workspace" 용어 충돌

① 좌측 세션 컨테이너(= 프로젝트 디렉토리), ② 우측 패널의 실행 표면(Run · Files · Terminal · Context). 같은 단어가 두 의미로 쓰이는 상태.

목표

작업의 성격별로 진입점을 나눠, 사용자가 "지금 뭐 하려는지"를 먼저 표현하고 들어오게 만든다. 각 진입점은 그 작업 한 가지에 최적화한다.

핵심 원칙

사용자가 도구의 구조에 맞춰 작업 방식을 바꾸지 않도록. 도구가 작업의 성격을 따라간다. 프로젝트 단위 지속 작업은 Project, 일회성 질문은 Ask ARIS, 시스템 전반 점검은 Home, 파일 탐색은 Files.

02 · Proposal overview

네 개의 표면, 하나의 좌측 레일.

좌측 사이드바를 New chat + 네 개 고정 진입점 + 프로젝트 목록으로 재구성합니다. 각 진입점은 뚜렷한 작업 성격에 대응하고, 진입점 간 이동은 한 번의 클릭으로 가능합니다.

네 진입점 요약

좌측 사이드바 구조

네 진입점은 사이드바 상단에 고정, 그 아래 프로젝트 목록. 진입점은 작업의 성격, 프로젝트는 작업의 대상이므로 시각적으로 분리합니다.

03 · Home

아침의 첫 화면.

로그인 직후 기본 랜딩. 지금 이 서버에서 무슨 일이 벌어지고 있는지 한 페이지로 요약합니다. 사용자는 여기서 오늘 뭐부터 할지를 결정하고 필요한 진입점으로 내려갑니다.

주요 결정

  • 채팅 시작 위치로 쓰지 않음. New chat은 사이드바에만, Home은 진행 상황 표시에 집중.
  • 서버 상태는 요약 4장 (Network I/O · CPU · Memory · Disk). ARIS는 기본적으로 원격 서버에 연결해 쓰는 시스템이라, 네트워크 스루풋을 최상위 지표로 배치. 상세 메트릭은 별도 Observability 뷰로 분리(open question).
  • 프로젝트 카드 = 진입점. 클릭 시 Project → Overview.
  • Activity feed는 agent + user 양쪽 섞어 표시 — 내 작업 + 에이전트 작업 통합 추적.

프로젝트 카드 내부 구조

카드에는 프로젝트의 최근 채팅 2건을 고정으로 노출합니다. LLM 요약이나 별도 summary 필드 없이, 이미 존재하는 채팅 데이터만 조합해 렌더하므로 생성 비용 0 · 항상 최신입니다.

각 채팅 행 = 세 조각

chat-screen-regressions
mobileOverflowLayout 버그 재현되는데, settle 루프에 가드 좀 추가해줘.
  • ① Status dot — running blue / completed green / needs-approval amber / idle gray. 채팅 자체의 상태.
  • ② 채팅 제목 — 1줄 ellipsis. 프로젝트 내 채팅 엔티티의 title 그대로.
  • ③ 마지막 사용자 메시지 — 1줄 ellipsis. 해당 채팅의 가장 최근 user role 메시지 본문.

데이터 생성 규칙

  • 각 프로젝트에 대해 updated_at DESC 2건을 뽑음. 단순 SQL·ORM 쿼리로 해결.
  • 채팅 상태는 이미 chat entity에 존재하는 필드(chat.status)를 그대로 바인딩.
  • 마지막 사용자 메시지는 messages WHERE chat_id=? AND role='user' ORDER BY created_at DESC LIMIT 1.
  • 채팅 2건 미만이면 남는 자리는 빈 행 placeholder 또는 영역 축소.
  • LLM 요약 · 캐시 · 백그라운드 잡 일절 없음.
Open question

정보 밀도가 높아 산만하다는 리스크. 초기 릴리스에선 카드 · 피드 수를 제한하고, Focus 모드(일부 섹션 숨김) 옵션 검토.

04 · Ask ARIS

프로젝트 없이, 질문부터.

"지금 뭐 하나만 물어볼래"의 경로. 프로젝트 선택 · 컨텍스트 설정 없이 바로 채팅이 열립니다. 동시에 과거에 나눈 모든 채팅을 전문 검색해 "그때 뭐라고 결정했더라?"를 빠르게 되살립니다.

핵심 특징

  • Project-less 채팅. 어느 프로젝트에도 속하지 않음. "Move to project"로 나중에 승격 가능.
  • 전역 채팅 검색. 키워드 입력 시 과거 모든 채팅 full-text 검색. 결과에 프로젝트 태그 + 매칭 스니펫 표시.
  • 인용 표시. 응답에 과거 채팅에서 가져온 정보가 있으면 [lawdigest · deploy flow] 같은 출처 태그 부착. 클릭 시 해당 채팅으로 점프.
  • Suggested prompts. 최근 작업 기반 제안. 초기 정적 템플릿, 이후 사용 패턴 학습.
Use cases

① 회고: "지난주 내가 내린 결정 중 제일 큰 건 뭐야?" → 최근 7일치 요약

② 지식 재활용: "예전에 tokens.css 마이그레이션 어떻게 했지?" → 해당 채팅 요약 + 코드 스니펫

③ 가벼운 Q&A: "React 19 메이저 변경점" → 외부 지식 답변 (컨텍스트 없이)

④ 명령어 찾기: "저번에 쓴 tail jq 명령어" → 과거 터미널 기록 검색

상태: empty / searching / conversing

Ask ARIS는 세 상태를 가집니다. ① Empty — Figure 5처럼 검색창 + 제안 + 최근 질문. ② Searching — 검색어 입력 시 과거 채팅 매칭 결과 드롭다운. ③ Conversing — 엔터 시 일반 채팅 뷰(composer · timeline)로 전환. Conversing 상태 UI는 chat-prototype.html과 동일 (모델 선택기 · 모드 토글 · 타임라인 전부 재사용).

05 · Project

프로젝트는 채팅보다 큰 단위다.

Project 진입점을 클릭하면 프로젝트 목록이 뜨고, 하나를 선택하면 Overview / Chats / Files / Context 탭을 가진 상세 뷰로 들어갑니다. 기존의 "Workspace → 채팅 리스트" 구조를 여기로 흡수하면서, 파일 · 컨텍스트 · 자산을 함께 볼 수 있게 확장했습니다.

목록 화면 결정 사항

  • 최근 채팅 2건은 Home 카드와 동일 컴포넌트 재사용. 별도 데이터 생성 없이 chats ORDER BY updated_at DESC LIMIT 2만 더 뽑음. Home과 일관성 유지.
  • 스탯 3개로 압축(Chats · Files · Tokens). 상세에서 보여주는 Last activity는 카드 푸터에 인라인으로 표시.
  • "New chat" 버튼은 카드 푸터에 노출. 프로젝트 진입을 거치지 않고 바로 채팅 시작 가능 — 빈도 높은 액션이라 한 단계 줄임.
  • "New project" 카드는 그리드 마지막 슬롯. 가산점이 아니라 이 화면의 부가 액션이므로 점선 카드로만.
  • 필터 칩 4종: All(기본) · Active(running/approval만) · Recent(7일 내 활동) · Archived. 상태 아이콘은 카드 우상단 배지로 일관.
  • 20개 넘으면 Active 그룹 자동 분리(open question에 명시된 정렬/필터 규칙의 첫 단계).

탭별 역할

  • Overview — 움직이는 일 + 고정 자산 한 화면. 재진입 시 "지금 어디까지 왔지"를 5초 안에 파악.
  • Chats — 이 프로젝트 채팅 리스트. 시간순 정렬 + 상태·키워드 필터.
  • Files — 프로젝트 디렉토리 파일 트리 + 최근 diff. 프로젝트 스코프로 제한.
  • Context — AGENTS.md · 스니펫 · 저장된 프롬프트 · 첨부 지식 파일.

Chats 탭의 구조

프로젝트로 이미 스코프가 좁아졌으니 시간순 그룹(Today/Yesterday)이 유의미. 채팅 카드에 상태 pill + 마지막 에이전트 응답 요약 표시.

채팅 카드 클릭 시 chat-prototype.html의 채팅 화면으로 이동. breadcrumb은 Project / aris-web / chat-screen-regressions.

06 · Files

서버 전체, 하나의 탐색기.

프로젝트 경계를 넘어서는 파일 작업을 위한 진입점. 서버 루트부터의 디렉토리 트리 · 검색 · 프리뷰 · 오픈 액션. 프로젝트 내부 파일 작업은 Project → Files 탭에서 하는 것이 자연스럽지만, 프로젝트 밖 파일(전역 스크립트 · 로그 · 백업)은 여기서 다룹니다.

결정

  • 3-pane 레이아웃. 트리 (Projects → System) · 리스트 (Name/Owner/Size/Modified) · 프리뷰.
  • Open in Project. 파일이 프로젝트 디렉토리 하위면, 해당 프로젝트 Files 탭으로 점프하며 파일 선택 상태.
  • Ask about this file. Ask ARIS 진입점으로 이동하며 파일을 첨부 + 질문 입력 상태.
  • 프로젝트 밖 파일도 질문 가능. 로그 · 설정 · 백업 같은 것도 즉석 분석.
Open question

깊은 디렉토리(node_modules · venv 등)를 기본으로 어떻게 가릴 것인가? .gitignore 기반 자동 필터 + 사용자 토글 필요.

07 · Shared UX

네 진입점을 관통하는 공통 규칙.

New chat 버튼의 의미는 진입점에 따라 달라진다

  • Home에서: 빈 상태 모달 — "어느 프로젝트에? Ask ARIS?" 선택.
  • Ask ARIS에서: project-less 채팅 즉시 시작.
  • Project에서: 해당 프로젝트 새 채팅 즉시 시작.
  • Files에서: Home과 동일한 선택 모달.

Breadcrumb은 어디 있는지 항상 알려준다

상단 topbar에 breadcrumb을 두고 현재 위치 표시. 예: Project / aris-web / Chats / chat-screen-regressions. 각 조각 클릭 시 해당 레벨로 되돌아감.

채팅 화면은 진입점과 무관하게 동일

Ask ARIS든 Project든 채팅 UI 자체는 chat-prototype.html과 동일 (타임라인 · composer · workspace pane · preview). 다른 건 breadcrumb과 컨텍스트 범위뿐.

상태 dot 색상 시스템

  • Running — 실행 중 에이전트가 있음.
  • Needs approval — 사용자 승인 대기.
  • Completed — 최근에 끝남.
  • Idle — 활동 없음.
08 · Migration

기존 데이터는 어디로?

IA를 바꾼다는 건 기존 Workspace → Chat 구조를 리패킹한다는 뜻. 모든 채팅이 1:1로 디렉토리에 묶인 Workspace에 속해 있어 매핑은 단순합니다.

단계별 진행

  1. 스키마 변경: workspace_id → project_id rename. 참조 업데이트.
  2. 백엔드 API 정리: /workspaces/* → /projects/* alias 추가, 기존 경로 deprecation. 6주 뒤 제거.
  3. 프론트 IA 교체: 사이드바 교체. ?ui=v1로 롤백 가능.
  4. Ask ARIS 출시: 전역 채팅 검색 인덱스(임베딩 + BM25). 초기 검색만, 채팅은 2차 릴리스.
  5. Home 출시: 프로젝트 카드 + Activity feed 우선. 서버 상태 strip은 모니터링 연결 필요.
  6. Files 출시: 파일 탐색기. 루트 디렉토리 이슈(node_modules · .git) 기본 필터.
사용자 영향

기존 사용자는 Workspace → Project 용어 변경만 경험. 채팅 위치 · 히스토리 · 상태는 유지. 새 기능은 추가적 가치로 등장.

주의

스크린샷 · 문서 · 튜토리얼의 "Workspace" 언급 전수 업데이트 필요. 레거시 링크 /workspaces/:id → /projects/:id 301 리다이렉트.

09 · Open questions

결정이 필요한 것들.

  1. Home의 정보 밀도 상한. 카드 · 피드 갯수 모바일 · 태블릿 기준 필요.
  2. Ask ARIS 검색 범위. 민감한 프로젝트는 제외 옵션? Project settings에 "Include in Ask ARIS index" 토글.
  3. 프로젝트 목록 정렬/필터. 20개 넘어가면 active · recent · archived 그룹핑 필요.
  4. Files ↔ Project Files 관계. "Open in Project" 반대 방향 필요성 검토.
  5. 채팅 이동. Ask ARIS 채팅을 프로젝트로 옮기는 UI 위치와 데이터 이전 정책.
  6. 권한 · 멀티 유저. Project가 팀 단위인지 개인 단위인지. Ask ARIS 검색 범위.
  7. 키보드 단축키. ⌘1~⌘4로 진입점 이동? ⌘K로 Ask?
